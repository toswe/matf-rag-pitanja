### **Pitanje 2.10. Testiranje u razvoju softvera. Cena greške u kontekstu vremena otkrivanja.**

**Odgovor:**

Cena greške u kontekstu vremena otkrivanja je **kritički koncept** u razvoju softvera koji pokazuje da je ispravljanje grešaka značajno jeftinije kada se otkrivaju u ranim fazama razvoja, a skupnije u kasnijim fazama.

Greške otkrivene u različitim fazama razvoja imaju sledeće troškove:

1. **Faza analize zahteva** - Najniži trošak. Greška zahteva samo reviziju definisanih zahteva i dodatno vreme analitičara.
2. **Faza dizajna softvera** - Nešto viši trošak. Ispravka podrazumeva izmenu dizajna, što zahteva dodatni rad dizajnera i ažuriranje opisa implementacije.
3. **Faza implementacije softvera** - Umeren trošak. Greške otkrivene tokom kodiranja obično programer brzo ispravi. Posebno je bitna faza integracije komponenti - greške ovde su teže za otkrivanje jer uključuju više različitih delova sistema. Ispravka zahteva saradnju više članova tima.
4. **Faza sistemskog testiranja** - Znatno viši trošak. Cena obuhvata rad tima za kvalitet softvera, prijavu greške, komunikaciju sa programerima, ponovljena testiranja i praćenje kroz sistem za praćenje defekata.
5. **Faza upotrebe (produkcija)** - Najveća cena. Pored tehničke ispravke, dolazi do gubitka poverenja korisnika i potencijalne reputacione štete.

**Zaključak:** Ranija otkrivanja greške znači brže, jednostavnije i jeftinije ispravljanje. To je razlog zašto je **testiranje tokom razvoja neophodna praksa**.

### **Pitanje 2.11. Testiranje u razvoju softvera. Uloga testera u razvoju softvera.**

**Odgovor:**

**Tester softvera** je stručnjak koji se bavi planiranjem, izvođenjem i dokumentovanjem aktivnosti testiranja softverskog sistema, sa ciljem da se obezbedi očekivani kvalitet proizvoda. Njegov osnovni zadatak je sistematsko ispitivanje softvera kako bi se identifikovali defekti u funkcionisanju, performansama ili upotrebljivosti.

**Organizacione strukture i raspodela obaveza:**

1. **Programeri kao testeri** - Programeri su istovremeno i testeri i sami proveravaju svoj kod (često u manjim projektima).
2. **Integrisan pristup** - Programeri i testeri rade zajedno u istom timu, blisko sarađujući.
3. **Odvojeni timovi** - Programeri i testeri čine potpuno odvojene timove sa jasnim razgraničenjem odgovornosti (često u većim projektima).

**Problemi u saradnji:**

Kada su programeri i testeri u razdvojenim timovima, često postoje problemi u lošoj komunikaciji, međusobnom nerazumevanju i opštoj netrpeljivosti. Ovi problemi su posledica **psihološke reakcije**:

- Za testera - pronalaženje greške znači da je uspešno obavio svoj zadatak
- Za programera - pronalaženje greške znači da nije uspešno obavio svoj posao

**Znanja i veštine testera:**

Tester mora da poseduje:

- Osnovno razumevanje programiranja i procesa razvoja softvera
- Detaljno poznavanje procedura i procesa testiranja
- Znanje o alatima koji se koriste u testiranju

**Zaključak:** Tester je ključan član razvoja softvera čija je primarna odgovornost sistematsko pronalaženje greške pre nego što softver stigne do korisnika.

### **Pitanje 2.12. Testiranje u razvoju softvera. Faze testiranja softvera: planiranje, analiza, dizajn i implementacija testova.**

**Odgovor:**

Testiranje softvera se odvija kroz jasno definisane faze:

**1. Faza planiranja (Planning)**

- Definiše se **strategija testiranja** i **plan testiranja**
- Identifikuju se resursi, vremenski okvir, budžet i alati koji će se koristiti
- Određuje se **obim testiranja** (koje delove sistema testirati)
- Definiše se **pristup testiranju** (statičko/dinamičko, manuelno/automatsko, itd.)
- Postavljaju se **ciljevi testiranja** i kriterijumi za završetak testiranja

**2. Faza analize (Analysis)**

- **Analiza zahteva** - detaljno razumevanje šta softver treba da radi
- **Analiza specifikacija** - pregled specifikacija sistema
- **Identifikacija test slučajeva** - određivanje šta treba testirati
- Identifikovanje rizika i kritičnih delova softvera
- Prioretizacija test slučajeva prema važnosti

**3. Faza dizajna (Design)**

- **Pisanje test slučajeva** - detaljno opisivanje kako se vrši testiranje
- Kreianje **test scenarija** - sekvensi testnih koraka
- Definisanje **očekivanih rezultata** za svaki test slučaj
- Kreiranje **test podataka** - specifični ulazni podaci za testiranje
- Organizovanje testova u logične grupe i prioritete

**4. Faza implementacije (Implementation)**

- **Pisanje test skriptova** - automatizovani testovi (ako se koristi automatizacija)
- Priprema **test okruženja** - instalacija, konfiguracija potrebnih alata i sistema
- Kreiranje **test okvira** (test framework) - zajedničku infrastrukturu za testove
- Organizovanje test slučajeva u **test suites** (kompleti testova)
- Verifikacija da je sve spremi za izvršavanje

**Primer:** Za bankarsku aplikaciju, u fazi planiranja bi se reklo “testiramo transfer sredstava”, u analizi bi se identifikovalo “trebamo testirati transfer između računa u istoj banci”, u dizajnu bi se napisalo “unesite 100 dinara na račun X, očekuje se uspešan transfer”, a u implementaciji bi se automatizovao test korišćenjem alata.

### **Pitanje 2.13. Testiranje u razvoju softvera. Faze testiranja softvera: izvršavanje i evaluacija testova. Zatvaranje testiranja.**

**Odgovor:**

Nakon što se testovi dizajniraju i implementiraju, slede faze izvršavanja, evaluacije i zatvaranja.

**1. Faza izvršavanja (Execution)**

- **Pokretanje test slučajeva** - izvršavanje testova prema redosledu iz plana
- **Prikupljanje rezultata** - beleženje šta se desilo pri izvršenju
- **Pronalaženje grešaka** - identifikovanje greške kada test ne prođe
- **Otvaranje prijava grešaka** - formalno evidentiranje svakog pronađenog problema
- **Praćenje napredovanja** - monitoring do kada je testiranje završeno

**Dva pristupa:**

- **Manuelno testiranje** - tester ručno izvršava svaki test
- **Automatizovano testiranje** - test skriptovi se izvršavaju automatski

**2. Faza evaluacije (Evaluation)**

- **Analiza rezultata testiranja** - pregled šta je prošlo, a šta nije
- **Izveštavanje o greškama** - detaljno dokumentovanje svih pronađenih problema
- **Prioritizacija grešaka** - određivanje redosleda ispravljanja (kritične prvo)
- **Praćenje ispravki** - verifikacija da su pronađene greške zaista ispravljene
- **Procena kvaliteta** - određivanje da li je softver spreman za puštanje u rad

**Kriterijumi za završetak testiranja:**

- Svi planiranitest slučajevi su izvršeni
- Sve kritične greške su ispravljene
- Postignuta je dovoljna pokrivenost koda (npr. 80%+)
- Nema Open (neispravljenih) grešaka visokog prioriteta
- Sve regresione greške su ispravljene

**3. Faza zatvaranja (Closure)**

- **Finalna verifikacija** - poslednja provera da je softver spreman
- **Dokumentovanje rezultata** - sveobuhvatan izveštaj o testiranju
- **Prenos znanja** - transfer informacija korisnicima i tim za podršku
- **Lekције izvučene** - analiza šta je bilo dobro, a šta se može poboljšati
- **Zatvaranje test artefakata** - čuvanje dokumenata za budućnost

**Primer:** Nakon što se bankarska aplikacija testira (faza izvršavanja), tester pravi izveštaj (evaluacija) da je pronašao 15 grešaka od kojih su 3 kritične. Kada se sve isprave, tester potvrđuje da je softver spreman za puštanje (zatvaranje), a tim naučila je koja je test strategija bila efikasna.

### **Pitanje 2.14. Vrste testiranja. Testiranje jedinca koda. Primeri.**

**Odgovor:**

**Testiranje jedinica koda (Unit Testing)** je vrsta testiranja u kojoj se **pojedinačne funkcije, metode ili male komponente koda testiraju u izolaciji**, bez zavisnosti od ostatka sistema.

**Karakteristike:**

- Najjednostavniji nivo testiranja
- Testira se najmanja moguća jedinica koda (funkcija, metoda, klasa)
- Obično ga vrši **programer koji je napisao kod**
- Koristi se **automatizovano testiranje** (test framework)
- Test je brz i lak za izvršenje
- Ima **visoku pokrivenost koda**
- Otkrivaju se greške **najranije**

**Primeri:**

1. **Testiranje funkcije za sabiranje (Python):**

```python
def saberi(a, b):
    return a + b

# Test
assert saberi(2, 3) == 5
assert saberi(-1, 1) == 0
assert saberi(0, 0) == 0
```

1. **Testiranje validacije email adrese:**

```python
def je_validna_email(email):
    return "@" in email and "." in email

# Test
assert je_validna_email("korisnik@gmail.com") == True
assert je_validna_email("pogresan_email") == False
```

1. **Testiranje funkcije za računanje kamate (finansijska aplikacija):**

```python
def izracunaj_kamatu(glavnica, stopa, vreme):
    return (glavnica * stopa * vreme) / 100

# Test
assert izracunaj_kamatu(100, 5, 1) == 5
assert izracunaj_kamatu(1000, 10, 2) == 200
```

1. **Testiranje metode za proveru da li je broj prost:**

```python
def je_prost_broj(n):
    if n < 2:
        return False
    for i in range(2, int(n**0.5) + 1):
        if n % i == 0:
            return False
    return True

# Test
assert je_prost_broj(2) == True
assert je_prost_broj(7) == True
assert je_prost_broj(10) == False
```

**Prednosti:**

- Otkrivaju se greške veoma rano
- Brzo se izvršavaju
- Lako se pronalaze bugovi u specifičnom kodu
- Omogućava **refaktoriziranje** bez straha

**Napomena:** Jedinično testiranje se obično sprovodi automatizovano korišćenjem test framework-a kao što su JUnit (Java), pytest (Python), NUnit (C#), itd.

### **Pitanje 2.15. Vrste testiranja. Komponentno i integraciono testiranje. Primeri.**

**Odgovor:**

**Komponentno testiranje** i **integraciono testiranje** su nivoi testiranja koji spajaju više jedinica koda.

**KOMPONENTNO TESTIRANJE (Component Testing)**

Testiranje **pojedinačnih komponenti** - većih blokova koda koji se sastoje od više jedinica (klase, moduli, paketi).

**Karakteristike:**

- Testira se komponenta u izolaciji, ali sa više funkcija/metoda
- Obično ga vrši **tester ili programer**
- Koristi se **automatizovano testiranje**
- Testa se interfejs komponente
- Zanemaruju se zavisnosti od drugih komponenti (koriste se mock objekti)

**Primer - Komponentno testiranje bankarskog modula za transfer:**

```
Komponenta: TransferModule
- Funkcija: validacija_iznosa()
- Funkcija: provera_dostupnosti_sredstava()
- Funkcija: izvrsavanje_transfera()

Test: Testiramo sve tri funkcije kao celinu, ali bez stvarne baze podataka
```

---

**INTEGRACIONO TESTIRANJE (Integration Testing)**

Testiranje **interakcije između više komponenti** kada se spajaju u veće celine.

**Karakteristike:**

- Testira kako komponente **funkcionišu zajedno**
- Pronalazi greške u komunikaciji između komponenti
- Obično ga vrši **tester**
- Koristi se **automatizovano ili manuelno testiranje**
- Otkrivaju se problemi sa integracijama, API-jima, bazama podataka

**Primeri integracionog testiranja:**

**Primer 1 - Bravia paket (iz literature):**
Komponenta 1: Login modul
Komponenta 2: Profil korisnika
Komponenta 3: Notifikacijski sistem

Integraciono testiranje: Kada se korisnik uloguje, da li se profil ispravno učita i da li se notifikacija pravilno prikazuje?

**Primer 2 - Bankarski sistem:**

- **Komponenta 1**: Modul za login
- **Komponenta 2**: Modul za transfer sredstava
- **Komponenta 3**: Modul za notifikacije

**Integraciono test:** Korisnik se loguje → izvršava transfer → prima notifikaciju. Testira se da li sve ove komponente rade zajedno.

**Primer 3 - E-commerce sistema:**

- **Komponenta 1**: Shopping Cart
- **Komponenta 2**: Payment Processing
- **Komponenta 3**: Inventory Management

**Test:** Korisnik dodaje proizvod u korpu → vrši plaćanje → sistem automatski smanjuje zalihe. Proveravamo da li se sve komponente sinhronizuju.

**Nivoi integracionog testiranja:**

1. **“Big Bang”** - sve komponente se integrišu odjednom (rizično)
2. **“Top-Down”** - počinje od vrha, gradimo prema dnu
3. **“Bottom-Up”** - počinje od dna, gradimo prema vrhu (češće)
4. **“Sandwich”** - kombinacija Top-Down i Bottom-Up

**Razlika između komponentnog i integracionog:**

- Komponento testiranje: Testira se jedan modul sa svim funkcijama
- Integraciono testiranje: Testira se kako se više modula međusobno podržava

### **Pitanje 2.16. Vrste testiranja. Sistemsko testiranje. Funkcionalno sistemsko testiranje. Regresiono testiranje. Primeri.**

**Odgovor:**

**SISTEMSKO TESTIRANJE** je testiranje **kompletnog sistema kao celine**, sa svim komponentama integrisanim.

---

**1. FUNKCIONALNO SISTEMSKO TESTIRANJE**

Testira se da li **sistem obavlja sve funkcionalnosti kako je specificirano**.

**Karakteristike:**

- Testira se **kompletan sistem** iz perspektive korisnika
- Provera da li sistem **ispunjava sve zahteve**
- Tester ignoriše internu strukturu koda
- Koristi se **manuelno i automatizovano testiranje**
- Testira se kroz **end-to-end scenarije**

**Primeri:**

**Primer 1 - Bankarsko mobilna aplikacija:**

- Test 1: Korisnik se loguje sa email i lozinkom - očekuje uspešnu prijavu
- Test 2: Korisnik vidi sve svoje račune na početnoj ekranu
- Test 3: Korisnik može preneti novac sa računa A na račun B - transfer je uspešan
- Test 4: Korisnik može platiti račun - plaćanje je potvrđeno
- Test 5: Korisnik može preuzeti izvod - izvod je dostupan kao PDF

**Primer 2 - E-commerce platforma:**

- Test: Korisnik prosledi kroz: pretraga proizvoda → dodavanje u korpu → plaćanje → porudžbina je potvrđena
- Test: Korisnik može promeniti količinu proizvoda u korpi
- Test: Sistem prikazuje tačnu cenu sa porezom i dostavljačkom geburi

---

**2. REGRESIONO TESTIRANJE**

Provera da **nove izmene nisu poremetile postojeće funkcionalnosti**.

**Karakteristike:**

- Testira se **stari kod** nakon što se novi kod doda
- Cilj: Osigurati da nema “bočnih efekata”
- Izvršavaju se **već postojeći testovi** koji su prethodno bili dobri
- **Obavezno automatizovano testiranje** (jer se često ponavljaju)
- Sprečava **regresiju** (vraćanje na starije greške)

**Primeri:**

**Primer 1 - Bankarsko aplikacija:**

- Originalni test: Transfer sredstava između računa funkcioniše
- Programer dodaje novu funkcionalnost: Mogućnost dodavanja beleške pri transferu
- Regresiono testiranje: Trebam da ponovo testiram originalni transfer da vidim da li je još uvek funkcioniše (bez beleške)

**Primer 2 - Gmail:**

- Originalno: Korisnik može slati email
- Nova funkcionalnost: Planiranje slanja email-a (send later)
- Regresiono testiranje: Trebam da testiram da normalno slanje email-a još radi (bez “send later”)

**Primer 3 - FIFA video igra:**

- Originalno: Igrač može driblovati loptu
- Nova funkcionalnost: Dodana nova specijalna finta
- Regresiono testiranje: Trebam da testiram da originalna dribla i sve ostale karakteristike još rade

**Kada se vrši regresiono testiranje:**

- Nakon ispravljanja greške
- Posle dodavanja nove funkcionalnosti
- Nakon ažuriranja biblioteke ili zavisnosti
- Pred svakim verzijama za puštanje

**Strategije regresionog testiranja:**

1. **Re-test all** - ponovi sve testove (skupo, ali pouzdano)
2. **Regression test selection** - ponovi samo relevantne testove
3. **Test case prioritization** - prioritizuj važnije testove

### **Pitanje 2.17. Vrste testiranja. Sistemsko testiranje. Istraživačko testiranje. Testovi prihvatljivosti. Instalaciono testiranje. Primeri.**

**Odgovor:**

---

**1. ISTRAŽIVAČKO TESTIRANJE (Exploratory Testing)**

**Karakteristike:**

- Tester **istražuje aplikaciju bez detaljnog plana**
- Nema unapred napisanih test slučajeva
- Tester kombinuje **testiranje sa učenjem**
- Akcenat je na **pronalaženju greške kroz eksperimentisanje**
- Tester koristi svoju intuiciju i iskustvo
- Odlično za pronalaženje **neočekivanih grešaka**
- Tipično se vrši **manuelno**

**Primeri:**

**Primer 1 - Testiranje nove mobilne aplikacije:**

- Tester preuzme aplikaciju i počne da je koristi
- Pokušava da unese karaktere u polje za broj - šta se dešava?
- Pokušava da ottvori aplikaciju sa loše internetskom vezom
- Pokušava da instala aplikaciju dva puta - šta se dešava?
- Pronalazi greške kroz slobodno istraživanje

**Primer 2 - Testiranje Gmail-a:**

- Tester prosledi kroz sve opcije na početnoj ekranu
- Pokušava da dodeli emejl sa više od 100 prilogama
- Pokušava da pošalje emejl sebi sa različitih vremenske zone
- Pronalazi nepredvidive greške kroz eksperimentisanje

---

**2. TESTOVI PRIHVATLJIVOSTI (Acceptance Testing)**

**Karakteristike:**

- Testira se da li **softver zadovoljava poslovne zahteve korisnika**
- Obavlja je **korisnik ili njegov predstavnik**
- Provera: **“Mogu li koristiti ovaj softver za svoj posao?”**
- Obavlja se **PRE nego što se softver pusti u rad**
- Ako prođe, softver se prihvata (acceptance)
- Ako ne prođe, softver se vraća na ispravku
- Koristi se **manuelno testiranje**

**Primeri:**

**Primer 1 - Bankarsko aplikacija:**
Klijent (banka) testira:

- Mogu li korisnici preneti novac između računa?
- Mogu li upravljati svoje račune?
- Kako izgleda izvod?
- Da li je brz sistem?

Ako klijent rekne “DA, PRIHVATA SE”, aplikacija ide u produkciju.

**Primer 2 - Sistem za prodaju ulaznica:**
Predstavnik kompanije koja prodaje karte testira:

- Mogu li da kreliram novu event?
- Mogu li da postavim cenu karata?
- Mogu li da vidim koliko je karata prodano?
- Sistem je spreman za upotrebu?

---

**3. INSTALACIONO TESTIRANJE (Installation Testing)**

**Karakteristike:**

- Testira se **proces instalacije softvera**
- Proverava se da li se softver ispravno instalira na različitim sistemima
- Testira se **deinstalacija** softvera
- Proverava se da li su svi potrebni fajlovi instalirani
- Testira se konfiguracija nakon instalacije
- Obavlja se **pre puštanja softvera korisnicima**
- Obavlja se **manuelno ili automatizovano**

**Primeri:**

**Primer 1 - Instalacija antivirus programa:**

- Preuzmi instalacijski program
- Instaliraj na čistom Windows sistemu
- Proveri da li se sve datoteke instaliraju
- Pokušaj deinstalirati
- Proveri da li su sve datoteke obrisane

**Primer 2 - Instalacija WordPress-a:**

- Preuzmem WordPress sa sajtа
- Instaliram na server
- Konfiguiram bazu podataka
- Proverim da li je sajt dostupan

**Primer 3 - Instalacija aplikacije na Android:**

- Preuzmem aplikaciju sa Google Play-a
- Instaliram na telefonu
- Proverm je li sve funkcionisalo pri prvom pokretanju
- Pokušam da deinstalram
- Proverm da li je memorija oslobođena

### **Pitanje 2.18. Vrste testiranja. Nefunkcionalno testiranje. Testovi performansi. Testovi kompatibilnosti. Testovi pouzdanosti. Primeri.**

**Odgovor:**

**NEFUNKCIONALNO TESTIRANJE** provera kako se sistem **ponaša**, a ne **šta radi**.

---

**1. TESTOVI PERFORMANSI (Performance Testing)**

**Karakteristike:**

- Testira se **brzina, odziv i kapacitet sistema**
- Provera kako sistem radi pod različitim opterećenjima
- Merenje: **vreme odziva, protok, korišćenje resursa**
- Obavlja se **automatizovano** (koriste se posebni alati)
- Kritično je za sisteme sa visokim saobraćajem

**Primeri:**

**Primer 1 - Kupovina avio karata:**

- Zahtev: Pretraga karata treba da se izvrši za manje od 2 sekunde u 97% slučajeva
- Test: Simulacija 10.000 korisnika koji istovremeno pretražuju karte
- Provera: Da li sistem odgovara u roku od 2 sekunde?

**Primer 2 - E-commerce Black Friday:**

- Zahtev: Sistem treba da bude dostupan kada posetilo više od 100.000 korisnika istovremeno
- Test: Simulacija 100.000 korisnika koji kupuju proizvode
- Provera: Koliko je korisnika moglo da završi kupovinu bez pada sistema?

**Primer 3 - Video streaming:**

- Zahtev: Vidoe trebaju da se učitavaju bez bufferisanja
- Test: Simulacija streaming-a sa različitim brzinama interneta (4G, LTE, Wi-Fi)
- Provera: Kolika je maksimalna rezolucija koju korisnik može da gleda bez prekida?

---

**2. TESTOVI KOMPATIBILNOSTI (Compatibility Testing)**

**Karakteristike:**

### **Pitanje 2.20. Tehnike testiranja. Karakteristike dobrog skupa testova. Pokrivenost testiranjem. Podela na tehnike testiranja.**

**Odgovor:**

---

**KARAKTERISTIKE DOBROG SKUPA TESTOVA**

Dobar skup testova ima sledeće karakteristike:

▶ Visokpotencĳal otkrivanja grešaka.
▶ Relativno mala veličina.
▶ Relativno velika brzina izvršavanja.
▶ Pružavisok stepen poverenja upouzdanost softvera.

‘’ iz skripte’’

---

**1. Kompletnost**

- Pokriva sve moguće scenarije (normalne, granične, pogrešne slučajeve)
- Svi zahtevi su testirani
- Nema zaboravljenih test slučajeva

**2. Validnost**

- Testovi proveravaju tačnu funkcionalnost
- Testovi su tačno napisani
- Očekivani rezultati su ispravni

**3. Nezavisnost**

- Testovi nisu međusobno zavisni
- Rezultat jednog testa ne utiče na drugi test
- Testovi se mogu izvršavati u bilo kojem redosledu

**4. Efikasnost**

- Testovi su brzo izvedivi
- Nema redundantnih testova (duplikata)
- Koriste se relevantni test podaci

**5. Jasnoća**

- Test slučajevi su jasno dokumentovani
- Očekivani rezultati su nedvosmisleni
- Lako se može razumeti šta se testira

---

**POKRIVENOST TESTIRANJEM (Test Coverage)**

**Definicija:** Procenat koda ili funkcionalnosti koji je obuhvaćen testovima.

Postoji : Pokrivenost zahteva, pokrivenost funkcionalnosti, pokrivenost koda

**Formule:**

- Test Coverage = (Broj pokrivenih redova koda / Ukupan broj redova koda) × 100%
- Test Coverage = (Broj pokrivenih funkcija / Ukupan broj funkcija) × 100%

**Primeri pokrivenosti:**

**Primer 1 - Python kod:**

```python
def je_pozitivan(broj):  # Linija 1
    if broj > 0:         # Linija 2
        return True      # Linija 3
    else:
        return False     # Linija 4
```

- Test samo sa pozitivnim brojem (5): Pokrivene linije 1,2,3 = 75% pokrivenost
- Test sa oba slučaja (5 i -5): Pokrivene linije 1,2,3,4 = 100% pokrivenost

Primer 2- Zahtev ≠ Funkcionalnost:

- Postoje zahtevi na specifikaciji
- Postoje funkcionalnosti koje se ne nalaze na specifikaciji ali su logican sled implementacije (dugme kupi ne radi kad je korpa prazna)
- U tim slucajevima funkcionalna pokrivenost i zahtevna pokrivenost nisu iste stvari

**Preporučeni nivoi pokrivenosti:**

- Minimalno: 60-70% (osnovna funkcionalnost)
- Normalno: 75-85% (dobar standard)
- Visoko: 85-95% (kritični sistemi)
- Veoma visoko: 95%+ (medicinski, vazduhoplovni sistemi)

---

**PODELA TEHNIKA TESTIRANJA**

Tehnike testiranja se mogu podeliti u tri glavne kategorije:

**1. TESTIRANJE CRNE KUTIJE (Black Box Testing)**

- **Šta se testira:** Ponašanje aplikacije bez znanja o internom kodu
- **Ko je testira:** Tester
- **Fokus:** Ulazni i izlazni podaci
- **Tehnike:**
    - Isprobavanje svih mogućih ulaza
    - Metod klasa ekvivalencije
    - Metod graničnih vrednosti
    - Tabele odlučivanja
    - Dijagrami stanja

**Primer:** Testiram banking aplikaciju tako što pokušavam da prenesem različite iznose i vidim da li se transferuje pravilno, bez gledanja kako kod funkcioniše.

---

**2. TESTIRANJE BELE KUTIJE (White Box Testing)**

- **Šta se testira:** Internu strukturu koda, logiku, putanje
- **Ko je testira:** Programer
- **Fokus:** Redovi koda, grane, putanje
- **Tehnike:**
    - Testiranje putanja
    - Pokrivenost redova koda
    - Pokrivenost grana
    - Pokrivenost iskaza

**Primer:** Testiranje banking aplikacije analizom kako kod obrađuje različite iznose - sve if iskaze, sve petlje.

---

**3. TESTIRANJE SIVE KUTIJE (Gray Box Testing)**

- **Šta se testira:** Kombinacija internog znanja i spoljnog testiranja
- **Ko je testira:** Tester koji ima osnovno znanje o kodu
- **Fokus:** Kombinacija logike i ponašanja

**Primer:** Testiram banking aplikaciju sa znanjem kako struktura baze podataka izgleda, ali fokusiram se na funkcionalnost.

---

Problem prorocista postoji kao situacija gde ne znamo sta je izlaz. Odnosno ne mozemo pretpostaviti.

### **Pitanje 2.21. Tehnike testiranja. Testiranje metodama crne kutije. Isprobavanja svih mogućih ulaza. Metod klasa ekvivalencije. Primeri.**

**Odgovor:**

---

**TESTIRANJE CRNE KUTIJE** - Tester ne zna kako kod funkcioniše, samo testira šta sistem radi.

---

**1. ISPROBAVANJA SVIH MOGUĆIH ULAZA (Exhaustive Testing)**

**Karakteristike:**

- Testira se **svi mogući ulazni podaci**
- Tester proba sve mogućnosti
- Teorijski: **Savršena pokrivenost**, u praksi: **Nemoguće za velike sisteme**
- Najjednostavniji pristup, ali nepraktičan

**Primer:**

Funkcija: `je_pozitivan(broj)` - proverava da li je broj pozitivan

Svi mogući ulazi: -∞, -1000, -100, -10, -1, 0, 1, 10, 100, 1000, ∞

Izlazi: True (pozitivan) ili False (nije pozitivan)

```
Testiranje svih mogućnosti:
je_pozitivan(-1000) → False ✓
je_pozitivan(-100)  → False ✓
je_pozitivan(-1)    → False ✓
je_pozitivan(0)     → False ✓
je_pozitivan(1)     → True ✓
je_pozitivan(100)   → True ✓
je_pozitivan(1000)  → True ✓
```

**Problem:** Za broj koji može biti od -∞ do +∞, nemoguće je testirati sve kombinacije!

---

**2. METOD KLASA EKVIVALENCIJE (Equivalence Class Partitioning)**

**Karakteristike:**

- Ulazni podaci se **dele na grupe (klase) sa sličnim ponašanjem**
- Testira se **jedan predstavnik iz svake klase**
- Smanjuje broj testova, povećava efikasnost
- Česti pristup u praksi

**Principi:**

1. Identifikuj klase ekvivalencije
2. Izaberi jedan test slučaj iz svake klase
3. Ponovi sve klase sa graničnim vrednostima

---

**PRIMERI:**

**Primer 1 - Funkcija `je_pozitivan(broj)`**

Klase ekvivalencije:

- Klasa 1: Negativni brojevi
- Klasa 2: Nula
- Klasa 3: Pozitivni brojevi

Testovi:

```
je_pozitivan(-5)    → False ✓  (predstavnik klase negativnih)
je_pozitivan(0)     → False ✓  (predstavnik klase nula)
je_pozitivan(5)     → True ✓   (predstavnik klase pozitivnih)
```

---

**Primer 2 - Polje za unos godišta**

Zahtev: “Unesi godinu između 1900 i 2024”

Klase ekvivalencije:

- Klasa 1: Prema mali broj (< 1900)
- Klasa 2: Validna godina (1900-2024)
- Klasa 3: Preveliki broj (> 2024)

Testovi:

```
Unos: 1800  → Greška "Godina premaljena!" ✓
Unos: 1950  → Prihvaceno ✓
Unos: 2030  → Greška "Godina premamala!" ✓
```

---

**Primer 3 - Sistem za kupovinu avio karata**

Zahtev: “Broj putnika: minimalno 1, maksimalno 9”

Klase ekvivalencije:

- Klasa 1: Prema mali (< 1)
- Klasa 2: Validna (1-9)
- Klasa 3: Prevelika (> 9)

Testovi:

```
Unos: 0  → Greška "Minimalno 1 putnik!" ✓
Unos: 4  → Prihvaceno - 4 putnika ✓
Unos: 15 → Greška "Maksimalno 9 putnika!" ✓
```

---

**Primer 4 - Polje email**

Zahtev: “Unesi validnu email adresu”

Klase ekvivalencije:

- Klasa 1: Validan email (sadrži @ i .)
- Klasa 2: Nevažeći email (nema @)
- Klasa 3: Nevažeći email (nema .)

Testovi:

```
Unos: korisnik@gmail.com    → Prihvaceno ✓
Unos: korisnikgmail.com     → Greška "Nevažeći format!" ✓
Unos: korisnik@com          → Greška "Nevažeći format!" ✓
```

---

**Prednosti klasa ekvivalencije:**

- ✓ Smanjuje broj testova
- ✓ Efikasno otkrivanje grešaka
- ✓ Jasna organizacija testova
- ✓ Lakše održavanje

**Nedostaci:**

- ✗ Može se propustiti greška na granicama klasa
- → To se rešava sa **metodom graničnih vrednosti** (što je sledeće pitanje)

### **Pitanje 2.22. Tehnike testiranja. Testiranje metodama crne kutije. Metod klasa ekvivalencije. Metod graničnih vrednosti. Primeri.**

**Odgovor:**

---

**METOD GRANIČNIH VREDNOSTI (Boundary Value Analysis)**

**Karakteristike:**

- Testira se **na granicama klasa ekvivalencije** gde se često javljaju greške
- Greške se često dešavaju na granama: između “manje od” i “veće od”
- Testira se **minimalnu i maksimalnu vrednost** svake klase
- Kombinuje se sa **metodom klasa ekvivalencije**

**Princip:**
Za interval od A do B, testiraju se vrednosti:

- A - 1 (malo ispod donje granice)
- A (donja granica)
- A + 1 (malo iznad donje granice)
- B - 1 (malo ispod gornje granice)
- B (gornja granica)
- B + 1 (malo iznad gornje granice)

---

**PRIMERI:**

**Primer 1 - Polje za unos godišta (1900-2024)**

Klase ekvivalencije:

- Klasa 1: < 1900
- Klasa 2: 1900-2024 (validna)
- Klasa 3: > 2024

Granične vrednosti:

```
Test: 1899  → Greška ✓         (granica - 1)
Test: 1900  → Prihvaceno ✓    (donja granica - validna)
Test: 1901  → Prihvaceno ✓    (granica + 1)
Test: 2023  → Prihvaceno ✓    (granica - 1)
Test: 2024  → Prihvaceno ✓    (gornja granica - validna)
Test: 2025  → Greška ✓         (granica + 1)
```

---

**Primer 2 - Broj putnika u sistemu za kupovinu karata (1-9)**

Klase ekvivalencije:

- Klasa 1: < 1
- Klasa 2: 1-9 (validno)
- Klasa 3: > 9

Granične vrednosti:

```
Test: 0   → Greška ✓       (< 1)
Test: 1   → Prihvaceno ✓   (minimalna validna vrednost)
Test: 2   → Prihvaceno ✓   (granica + 1)
Test: 8   → Prihvaceno ✓   (granica - 1)
Test: 9   → Prihvaceno ✓   (maksimalna validna vrednost)
Test: 10  → Greška ✓       (> 9)
```

---

**Primer 3 - Proveravanje uzrasta za vozačku dozvolu (18-65)**

Klase ekvivalencije:

- Klasa 1: < 18
- Klasa 2: 18-65 (validno)
- Klasa 3: > 65

Granične vrednosti:

```
Test: 17  → Greška "Preminuli ste!" ✓       (< 18)
Test: 18  → Prihvaceno ✓                     (minimalna validna)
Test: 19  → Prihvaceno ✓                     (granica + 1)
Test: 64  → Prihvaceno ✓                     (granica - 1)
Test: 65  → Prihvaceno ✓                     (maksimalna validna)
Test: 66  → Greška "Priprenostari!" ✓       (> 65)
```

---

**Primer 4 - Popust za kupovinu (10% popust za kupovine > 1000 dinara)**

Klase ekvivalencije:

- Klasa 1: <= 1000 dinara (bez popusta)
- Klasa 2: > 1000 dinara (sa 10% popusta)

Granične vrednosti:

```
Test: 999  → Cena bez popusta ✓
Test: 1000 → Cena bez popusta ✓     (granica)
Test: 1001 → Cena sa 10% popusta ✓  (granica + 1)
```

---

**Kombinovanje klasa ekvivalencije i graničnih vrednosti:**

**Primer 5 - Skup testova za bankarsku aplikaciju (transfer sredstava)**

Zahtev: “Unesite iznos za transfer: minimalno 10 dinara, maksimalno 100.000 dinara”

```
Klase ekvivalencije:
1. Manji od 10 dinara
2. Od 10 do 100.000 dinara (validno)
3. Veće od 100.000 dinara

Granične vrednosti - Set testova:

Test 1: 0 dinara         → Greška "Minimalno 10 dinara!" ✓
Test 2: 9 dinara         → Greška "Minimalno 10 dinara!" ✓
Test 3: 10 dinara        → Prihvaceno ✓              (donja granica)
Test 4: 50.000 dinara    → Prihvaceno ✓              (sredina - klasa 2)
Test 5: 100.000 dinara   → Prihvaceno ✓              (gornja granica)
Test 6: 100.001 dinara   → Greška "Maksimalno 100.000!" ✓
Test 7: 1.000.000 dinara → Greška "Maksimalno 100.000!" ✓
```

---

**Metod sa dve vrednosti:** Testiraju se samo donja i gornja granica:

```
Test: 10      → Prihvaceno ✓
Test: 100.000 → Prihvaceno ✓
```

**Metod sa tri vrednosti:** Testiraju se -1, granična vrednost, i +1:

```
Test: 9       → Greška ✓
Test: 10      → Prihvaceno ✓
Test: 11      → Prihvaceno ✓
```

**Metod sa pet vrednosti:** Kompletne granice:

```
Test: 9       → Greška ✓
Test: 10      → Prihvaceno ✓
Test: 99.999  → Prihvaceno ✓
Test: 100.000 → Prihvaceno ✓
Test: 100.001 → Greška ✓
```

---

**Prednosti graničnih vrednosti:**

- ✓ Pronalazi greške na granicama
- ✓ Brzo otkrivanje off-by-one grešaka
- ✓ Minimalan broj testova
- ✓ Efikasno u praksi

### **Pitanje 2.23. Tehnike testiranja. Karakteristike dobrog skupa testova. Tabele odlučivanja. Primeri.**

**Odgovor:**

---

**TABELE ODLUČIVANJA (Decision Tables)**

**Karakteristike:**

- Koristi se za testiranje **sistema sa složenim logikom i kombinacijama ulaza**
- Tabela prikazuje **sve kombinacije uslova i očekivane rezultate**
- Svaki red je **jedan test slučaj**
- Odličan metod za **pravila i politike**

**Struktura tabele:**

```
┌─────────────────────────────────────────────────────┐
│ Uslovi (Conditions)                                 │
├─────────────────────────────────────────────────────┤
│ Akcije (Actions)                                    │
└─────────────────────────────────────────────────────┘
```

---

**PRIMERI:**

**Primer 1 - Odobravaju kredita**

Zahtev: “Odobri kredit ako je korisnik star > 21 god I ima dobru kreditnu istoriju I godišnji prihod > 30.000 dinara”

```
┌──────────────────────────────────────────────────────────────┐
│ Test Case    │ Test 1 │ Test 2 │ Test 3 │ Test 4 │ Test 5  │
├──────────────────────────────────────────────────────────────┤
│ Uslovi:      │        │        │        │        │         │
│ Uzrast > 21  │  DA    │  DA    │  DA    │  NE    │  DA     │
│ Dobra hist.  │  DA    │  DA    │  NE    │  DA    │  DA     │
│ Prihod > 30k │  DA    │  NE    │  DA    │  DA    │  DA     │
├──────────────────────────────────────────────────────────────┤
│ Akcija:      │        │        │        │        │         │
│ Odobri       │ ✓      │ ✗      │ ✗      │ ✗      │ ✓       │
└──────────────────────────────────────────────────────────────┘
```

Testovi:

```
Test 1: Uzrast 30, dobra istorija, prihod 50k  → Odobren ✓
Test 2: Uzrast 30, dobra istorija, prihod 20k  → Odbijen ✓
Test 3: Uzrast 30, loša istorija, prihod 50k   → Odbijen ✓
Test 4: Uzrast 18, dobra istorija, prihod 50k  → Odbijen ✓
Test 5: Uzrast 30, dobra istorija, prihod 50k  → Odobren ✓
```

---

**Primer 2 - Popust za kupovinu**

Zahtev:

### **Pitanje 2.25. Tehnike testiranja. Testiranje metodama bele kutije. Putanja i broj putanja u programu. Pojam i vrste pokrivenosti. Njihovi odnosi. Primeri.**

**Odgovor:**

---

**TESTIRANJE BELE KUTIJE (White Box Testing)**

Tester **poznaje internu strukturu koda** i testira logiku, putanje, iskaze.

---

**PUTANJA U PROGRAMU (Program Path)**

**Definicija:** Sekvenca iskaza koja se izvršava od početka do kraja programa.

**Primer - Kod sa grananjem:**

```python
def provera_starosti(uzrast):     # Početak
    if uzrast >= 18:              # Iskaz 1 (If iskaz)
        print("Punoletan")         # Iskaz 2
        return True
    else:
        print("Nepunoletan")       # Iskaz 3
        return False
                                   # Kraj
```

Moguće putanje:

- **Putanja 1:** Početak → Iskaz 1 (true) → Iskaz 2 → Kraj
- **Putanja 2:** Početak → Iskaz 1 (false) → Iskaz 3 → Kraj

---

**BROJ PUTANJA (Path Count)**

Računanje broja mogućih putanja korišćenjem **ciklomatske složenosti**:

*C* = *E* − *N* + 2*P*

Gde je:

- E = broj grana (edges)
- N = broj čvorova (nodes)
- P = broj povezanih komponenti (obično 1)

---

**VRSTE POKRIVENOSTI**

**1. Pokrivenost iskaza (Statement Coverage)**

- **Cilj:** Svaki iskaz se izvrši najmanje jednom
- **Procenat:** (Broj izvršenih iskaza / Ukupan broj iskaza) × 100%

```python
def provera_starosti(uzrast):
    if uzrast >= 18:          # Iskaz 1
        print("Punoletan")     # Iskaz 2 ← Mora se izvršiti
        return True
    else:
        print("Nepunoletan")   # Iskaz 3 ← Mora se izvršiti
        return False
```

Test sa 100% pokrivenosti iskaza:

```
Test 1: uzrast = 20 → Iskazi 1,2 se izvršavaju
Test 2: uzrast = 15 → Iskazi 1,3 se izvršavaju
Pokrivenost: 100% (svi iskazi 1,2,3 su pokriveni)
```

---

**2. Pokrivenost grana (Branch Coverage)**

- **Cilj:** Sve grane IF iskaza se izvršavaju (true I false)
- **Procenat:** (Broj testiranih grana / Ukupan broj grana) × 100%

```python
if uzrast >= 18:        # Grana 1 (true)
    # ...
else:                   # Grana 2 (false)
    # ...
```

Test sa 100% pokrivenosti grana:

```
Test 1: uzrast = 20 → Grana 1 (true) se testira ✓
Test 2: uzrast = 15 → Grana 2 (false) se testira ✓
```

---

**3. Pokrivenost puta (Path Coverage)**

- **Cilj:** Sve moguće kombinacije putanja se testiraju
- **Procenat:** (Broj testiranih putanja / Ukupan broj putanja) × 100%

```python
def zahtevaj_pristup(je_admin, ima_dozvolu):  # Početak
    if je_admin:                               # Iskaz 1
        if ima_dozvolu:                        # Iskaz 2
            return True                        # Iskaz 3
        else:
            return False                       # Iskaz 4
    else:
        return False                           # Iskaz 5
                                               # Kraj
```

Moguće putanje:

- Putanja 1: je_admin=true, ima_dozvolu=true → True
- Putanja 2: je_admin=true, ima_dozvolu=false → False
- Putanja 3: je_admin=false, ima_dozvolu=bilo → False

Test sa 100% pokrivenosti puta:

```
Test 1: je_admin=true,  ima_dozvolu=true   → True ✓
Test 2: je_admin=true,  ima_dozvolu=false  → False ✓
Test 3: je_admin=false, ima_dozvolu=bilo   → False ✓
```

1. Pokrivenost uslova - meri da li je **svaki pojedinačni logički uslov** u izrazu (svaka booleova pod‑ekspresija) bio bar jednom i true i false.
- Primer:

```jsx
if (A && B) {
	//..
}

Uslovi : A true B true i A false B false zadovoljavaju (2) 100% pokrivenost

```

- Svaki jednostavan uslov (npr. **`A > 0`**, **`B < 5`**) mora bar jednom biti true i bar jednom false u nekom testu.
- Nije bitno da li su sve kombinacije uslova pokrivene, samo da je svaki uslov pojedinačno video i true i false.
1. Pokrivenost visestrukih uslova - znači da su **sve kombinacije** istinitosnih vrednosti svih jednostavnih uslova u jednoj odluci testirane bar jednom
- Primer

```jsx
if (A && B) {
    // ...
}
Uslovi: A, B → 4 kombinacije: (T,T), (T,F), (F,T), (F,F). (100 % pokirvenost)
MCC traži test set koji obuhvata sve ove kombinacije, 
ne samo da je if jednom true, jednom false.

```

- Posmatra sve jednostavne uslove u odluci **zajedno** i traži da se pokriju sve kombinacije njihovih vrednosti (T/F) u toj odluci.
- Za n uslova u jednom if‑u, MCC teoretski traži 2n2*n* kombinacija, pa je mnogo jači (strožiji) kriterijum od obične condition coverage.
1. Pokrivenost funkcija. - mera koja pokaz¨je koliki je procenat funkcija izvrsen

**ODNOSI POKRIVENOSTI**

Hierarchija pokrivenosti (od najmanje ka najvećoj):

```
Pokrivenosti funkcija < Pokrivenost iskaza/naredbi < Pokrivenost grana/odluka < Pokrivenost puta
```

```jsx
Pokrivenost grana/odluka < Pokrivenost uslova < Pokrivenost visestrukih uslova
```

**Odnos:**

- Ako je **pokrivenost puta 100%**, tada je i **pokrivenost grana 100%** (i iskaza 100%)
- Ako je **pokrivenost grana 100%**, tada je i **pokrivenost iskaza 100%**
- Ali OBRNUTO NE VAŽI!

**Primeri:**

**Primer 1:**

```python
if A:
    x = 1
else:
    x = 2
print(x)
```

- Pokrivenost iskaza: 100% sa samo 1 test (jer se print UVEK izvršava)
- Pokrivenost grana: Trebaju 2 testa (true i false grana if iskaza)

---

**Primer 2 - Kod sa dve if iskaze:**

```python
if uzrast >= 18:           # If 1
    status = "Odraslo"
if grad == "Beograd":      # If 2
    bonus = 100
```

- Pokrivenost iskaza: Može biti 100% sa 1 testom (ako se oba if-a izvrše)
- Pokrivenost grana: Trebaju 4 testa:
    - uzrast >= 18: true i false
    - grad == “Beograd”: true i false

- Testira se **svaka moguća putanja** kroz if iskaze
- Manje testova nego path coverage, ali /pouzdano
- **McCabe-ov pristup**

**Koraci:**

1. Kreiraj graf toka kontrole
2. Izračunaj ciklomatsku složenost (broj podstawnih putanja)
3. Kreiraj test slučaj za svaku put aniju

**Formula ciklomatske složenosti:**

*M* = *E* − *N* + 2*P*

Broj osnovnih putanja = M

---

**Primer 1 - Osnovne putanja**

```python
def provera_kredita(uzrast, prihod, istorija):  # Node 1
    if uzrast >= 18:                             # Node 2
        if prihod >= 30000:                      # Node 3
            if istorija == "dobra":              # Node 4
                return "ODOBREN"                 # Node 5
            else:
                return "ODBIJEN"                 # Node 6
        else:
            return "ODBIJEN"                     # Node 7
    else:
        return "ODBIJEN"                         # Node 8
                                                 # Node 9 (Kraj)
```

**Graf toka:**

```
        1
        |
        2 (uzrast >= 18)
       / \
      3   8
     / \
    4   7
   / \
  5   6
   \ / \
    9
```

Ciklomatska složenost:

- Grane (E): 8
- Čvorovi (N): 9
- P = 1
- M = 8 - 9 + 2(1) = 1

**Osnovne putanje (3 putanje za 2 uslova):**

1. Putanja 1: uzrast=25, prihod=50000, istorija=“dobra” → ODOBREN
2. Putanja 2: uzrast=25, prihod=50000, istorija=“loša” → ODBIJEN
3. Putanja 3: uzrast=25, prihod=20000, bilo šta → ODBIJEN
4. Putanja 4: uzrast=15, bilo šta, bilo šta → ODBIJEN

---

**2. TESTIRANJE NA OSNOVU GRAFA TOKA PODATAKA (Data Flow Testing)**

**Karakteristike:**

- Prati **kako se varijable definišu i koriste** u kodu
- Testira **def-use parove** (gde je varijabla definisana i gde se koristi)
- Pronalazi greške sa **neinicijalizovanim varijablama**

**Terminologija:**

- **DEF (Definicija):** Gde se varijabla prvo dodeluje vrednost
- **USE (Upotreba):** Gde se varijabla koristi (čita)

---

**Primer 2 - Def-Use graf**

```python
def izracunaj_popust(cena, vip):     # Početak
    popust = 0                        # DEF: popust = 0

    if vip:                           # USE: vip
        popust = 0.2                  # DEF: popust = 0.2
    else:
        popust = 0.1                  # DEF: popust = 0.1

    finalna_cena = cena * (1 - popust)  # USE: popust, cena; DEF: finalna_cena

    return finalna_cena               # USE: finalna_cena
```

**Def-Use parovi:**

1. (DEF popust na 0) → (USE u računanju finalne_cene)
2. (DEF popust na 0.2) → (USE u računanju finalne_cene)
3. (DEF popust na 0.1) → (USE u računanju finalne_cene)
4. (DEF finalna_cena) → (USE u return iskazu)

**Test slučajevi:**

```
Test 1: cena=1000, vip=true
- Putanja: popust=0.2 → finalna_cena=800 ✓

Test 2: cena=1000, vip=false
- Putanja: popust=0.1 → finalna_cena=900 ✓
```

---

**Primer 3 - Pronalaženje grešaka sa neinicijalizovanim varijablama**

```python
def kreiraj_porudzbinu(korisnik_id):
    if korisnik_id > 0:
        status = "NOVO"               # DEF: status
    # else: status nICE INICIJALIZOANA!

    print(status)                     # USE: status ← GREŠKA! Može biti neinicijalizovana

    return status
```

Def-Use analiza bi otkrila: **status se koristi bez inicijalizacije u else grani!**

Ispravljeni kod:

```python
def kreiraj_porudzbinu(korisnik_id):
    status = "NEPOZNAT"               # DEF: inicijalizacija za sve slučajeve

    if korisnik_id > 0:
        status = "NOVO"               # DEF: status

    print(status)                     # USE: status ✓ Sigurno inicijalizovana

    return status
```

---

**STRATEGIJE ZA IZBOR PUTANJA**

**1. All-Definitions (def-coverage)**

- Pokriviti svaki DEF bar sa jednom USE-om

**2. All-Uses (use-coverage)**

- Pokriviti svaki USE bar sa jednom DEF-om

**3. All-du-paths**

- Pokriviti svaki def-use put

def obradi_narudzbinu(kolicina, cena, ima_popust):  # Početak

    if ima_popust:
        ukupna_cena = ukupna_cena * 0.9              # DEF: ukupna_cena

    provizija = ukupna_cena * 0.05                   # USE: ukupna_cena; DEF: provizija

### **Pitanje 2.28. Tehnike testiranja. Problem proročišta. Metamorfno testiranje. Primeri.**

**Odgovor:**

---

**PROBLEM PROROČIŠTA (Oracle Problem)**

**Definicija:** Problem određivanja **očekivanog rezultata** kada se test izvršava.

**Primer problema:**

```python
def izracunaj_kvadratni_koren(broj):
    return broj ** 0.5

# Test
rezultat = izracunaj_kvadratni_koren(16)

# Koji je očekivani rezultat? 4 ili 3.9999999 ili 4.0000001?
# Kako znamo da je rezultat ispravan?
```

Problemi:

- Nema specifikacije očekivanog rezultata
- Rezultat je matematički neprecizna vrednost
- Teško je znati je li rezultat prihvatljiv

---

**METAMORFNO TESTIRANJE (Metamorphic Testing)**

**Karakteristike:**

- **Ne testira se očekivani izlaz direktno**
- Umesto toga, testira se **odnos između više izlaza** (metamorfne relacije)
- Rešava problem nedostatka Oracle-a
- Koristi se za sisteme gde je teško odrediti točan rezultat

**Princip:**

```
Umesto: Test input → Output → Provera sa očekivanim
Koristi: Test input 1 → Output 1
         Test input 2 (modifikovani) → Output 2
         Provera odnosa između Output1 i Output2
```

---

**PRIMERI METAMORFNIH RELACIJA**

**Primer 1 - Kvadratni koren**

```python
def kvadratni_koren(broj):
    return broj ** 0.5

# Metamorfna relacija: ako je x1 < x2, tada sqrt(x1) < sqrt(x2)

# Test bez Oracle-a:
x1 = 16
x2 = 25

sqrt_x1 = kvadratni_koren(x1)  # = 4
sqrt_x2 = kvadratni_koren(x2)  # = 5

# Provera odnosa:
assert sqrt_x1 < sqrt_x2  # 4 < 5 ✓
# Ne trebam da znam tačan rezultat, samo odnos!
```

---

**Primer 2 - Sortiranje**

```python
def sortiraj(lista):
    return sorted(lista)

# Metamorfna relacija: ako sortiramo listu dva puta, rezultat je isti

lista = [3, 1, 4, 1, 5]
sortirano_1 = sortiraj(lista)       # [1, 1, 3, 4, 5]
sortirano_2 = sortiraj(sortirano_1) # [1, 1, 3, 4, 5]

# Provera:
assert sortirano_1 == sortirano_2  # Idempotentnost ✓
```

---

**Primer 3 - Kalkulator**

```python
def saberi(a, b):
    return a + b

# Metamorfna relacija: sabiranje je komutativno
# saberi(a, b) == saberi(b, a)

a = 10
b = 20

rezultat_1 = saberi(a, b)   # 30
rezultat_2 = saberi(b, a)   # 30

assert rezultat_1 == rezultat_2  # 30 == 30 ✓
```

---

**Primer 4 - Pronalaženje maksimalne vrednosti**

```python
def maks(a, b):
    return a if a > b else b

# Metamorfna relacija: ako dodamo istu vrednost oba argumenta,
# rezultat se takođe povećava za tu vrednost

# Test 1:
a1, b1 = 5, 3
max1 = maks(a1, b1)  # 5

# Test 2 (povećaj za 10):
a2, b2 = 15, 13
max2 = maks(a2, b2)  # 15

# Provera:
assert max2 == max1 + 10  # 15 == 5 + 10 ✓
```

---

**Primer 5 - Konverzija temperatura**

```python
def celsius_u_fahrenheit(c):
    return c * 9/5 + 32

# Metamorfna relacija: ako je T1 > T2, tada F(T1) > F(T2)

c1 = 0
c2 = 100

f1 = celsius_u_fahrenheit(c1)   # 32
f2 = celsius_u_fahrenheit(c2)   # 212

assert f1 < f2  # 32 < 212 ✓
# Bez znanja tačnih vrednosti, znamo da je f1 < f2!
```

---

**Primer 6 - Pronalaženje medijane**

```python
def medijana(lista):
    sortirana = sorted(lista)
    n = len(sortirana)
    return sortirana[n // 2] if n % 2 == 1 else (sortirana[n//2-1] + sortirana[n//2]) / 2

# Metamorfna relacija: Permutacija ulaza ne menja rezultat

lista1 = [3, 1, 4, 1, 5, 9, 2]
lista2 = [5, 1, 9, 2, 3, 4, 1]  # Ista lista, drugačij redosled

med1 = medijana(lista1)
med2 = medijana(lista2)

assert med1 == med2  # Redosled ne utiče ✓
```

---

**METAMORFNE RELACIJE U PRAKSI**

| Problem | Metamorfna relacija |
| --- | --- |
| Sortiranje | Sortiraj(Sortiraj(X)) == Sortiraj(X) |
| Min/Max | Ako X < Y, Min(X,Y) == X |
| Uređene liste | Ako je lista sortirana, search(X) je brz |
| Simetrični algoritmi | F(X, Y) = F(Y, X) |
| Linearne funkcije | F(2X) = 2*F(X) |
| Konverzije | C_to_F(C_to_F_inv(X)) == X |

---

**PREDNOSTI METAMORFNOG TESTIRANJA**

- ✓ Rešava problem nedostatka Oracle-a
- ✓ Pronalazi greške bez znanja tačnog rezultata
- ✓ Čini testiranje lakšim za kompleksne sisteme
- ✓ Automatizovano testiranje mogućnosti

**Nedostaci:**

- ✗ Trebalo znati metamorfne relacije za svaki sistem
- ✗ Ne pronalazi sve greške
- ✗ Zahteva detaljno razumevanje problema

**2. Property-Based Testing**

Definiš **svojstva koja će biti uvek istinita**, onda test generiše podatke.

```python
from hypothesis import given, strategies as st

@given(st.lists(st.integers()))
def test_sortiraj(lista):
    """Generiši nasumične liste i testiraj sortiranje"""
    sortirano = sorted(lista)

    # Svojstvo 1: Sortirana lista je duža ili jednaka originalnoj
    assert len(sortirano) == len(lista)

    # Svojstvo 2: Svaki element je manji od sledećeg
    for i in range(len(sortirano) - 1):
        assert sortirano[i] <= sortirano[i + 1]

    # Svojstvo 3: Sortirana lista sadrži sve originalne elemente
    assert sorted(sortirano) == sorted(lista)
```

**Alati:**

- Hypothesis (Python)
- QuickCheck (Haskell)
- ScalaCheck (Scala)

**3. Symbolic Execution**

Analizira kod **simbolički** da pronađe sve moguće putanje.

```python
# Kod:
def test(x, y):
    if x > 5:
        if y < 3:
            return "Greška"
    return "OK"

# Symbolic Execution bi generisao:
# Test 1: x=6, y=2  → Greška
# Test 2: x=6, y=4  → OK
# Test 3: x=3, y=2  → OK
```

**4. Mutations (Mutation Testing)**

Generiše **mutantne verzije koda** da vidi da li testovi pronalaze greške.

```python
# Originalni kod:
def je_pozitivan(broj):
    return broj > 0

# Mutacija 1:
def je_pozitivan_mut1(broj):
    return broj >= 0  # Promenjena operacija

# Mutacija 2:
def je_pozitivan_mut2(broj):
    return broj < 0   # Obrnuta logika

# Test da proverite da li pronalazi mutacije:
test_je_pozitivan()  # Trebalo bi da prosledi originalni kod ali padne na mutantima
```

**Alati:**

- Stryker (JavaScript, Java, Python)
- PIT (Java)

**Primer 2 - Property-Based Testing sa Hypothesis**

```python
from hypothesis import given, strategies as st

@given(st.lists(st.integers()))
def test_max(lista):
    """Generise nasumične liste i testira max()"""
    if len(lista) > 0:
        maks = max(lista)
        # Svojstvo: max mora biti >= od svih elemenata
        for elem in lista:
            assert maks >= elem

# Hypothesis će generisati:
# [], [^1], [5, 3, 2], [-100, 50], ...
# I testirati sve!
```
